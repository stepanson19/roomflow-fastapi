# Подробный план видеопрезентации RoomFlow

Цель видео - за 5-7 минут показать, что проект соответствует требованиям: есть понятная задача, рабочий REST API, авторизация, CRUD, бизнес-алгоритм, тесты и pylint.

## Перед записью

1. Закрыть лишние окна, уведомления, мессенджеры и вкладки.
2. Открыть три окна:
   - браузер со Swagger: `http://127.0.0.1:8000/docs`
   - редактор кода с файлом `app/services/recommendations.py`
   - терминал в папке проекта
3. Если локальный сервер не запущен, выполнить:

```bash
source .venv/bin/activate
export ROOMFLOW_SECRET_KEY="local-dev-secret-change-me"
export ROOMFLOW_SEED_DEMO=1
export ROOMFLOW_ADMIN_EMAIL="admin@example.com"
export ROOMFLOW_ADMIN_PASSWORD="StrongPass123"
uvicorn app.main:app --reload
```

4. Проверить, что открывается `http://127.0.0.1:8000/docs`.
5. Данные для входа:
   - username: `admin@example.com`
   - password: `StrongPass123`
6. Поля `client_id` и `client_secret` в Swagger оставлять пустыми.

## Структура видео

### 0:00-0:40 - Вступление

Что говорить:

> Я представляю проект RoomFlow. Это REST-сервис для небольшой студии репетиций. В такой студии есть комнаты, оборудование, клиенты и бронирования. Основная задача сервиса - не просто хранить данные, а помогать подобрать подходящую свободную комнату по времени, бюджету, вместимости и нужному оборудованию.

Что показать:

1. Главную страницу `http://127.0.0.1:8000/`.
2. Коротко показать кнопки `Swagger UI`, `ReDoc`, `OpenAPI JSON`.
3. Перейти в `Swagger UI`.

### 0:40-1:35 - Авторизация

Что нажимать:

1. В Swagger нажать зеленую кнопку `Authorize` сверху справа.
2. В поле `username` ввести `admin@example.com`.
3. В поле `password` ввести `StrongPass123`.
4. `client_id` и `client_secret` не заполнять.
5. Нажать `Authorize`, затем `Close`.

Что говорить:

> В проекте реализована OAuth2/JWT-авторизация. После входа Swagger автоматически добавляет токен к защищенным запросам. Первый пользователь системы является администратором, поэтому он может управлять каталогом комнат и оборудования.

Проверка:

1. Открыть `GET /api/v1/auth/me`.
2. Нажать `Try it out`.
3. Нажать `Execute`.
4. Показать ответ с email и ролью `admin`.

Что говорить:

> Этот запрос показывает текущего пользователя. Он защищен токеном, то есть без авторизации его выполнить нельзя.

### 1:35-2:35 - Каталог комнат и оборудования

Что нажимать:

1. Открыть `GET /api/v1/equipment`.
2. Нажать `Try it out` -> `Execute`.
3. Показать список оборудования.
4. Открыть `GET /api/v1/rooms`.
5. Нажать `Try it out` -> `Execute`.
6. Показать комнаты `Velvet Room`, `Neon Garage`, `Loft Stage`.

Что говорить:

> Здесь видно, что база данных реляционная. Есть отдельные таблицы пользователей, комнат, оборудования, бронирований и связующая таблица между комнатами и оборудованием. В ответе комнаты сразу видно вложенное оборудование, потому что сущности связаны.

Если нужно показать создание:

1. Открыть `POST /api/v1/equipment`.
2. Нажать `Try it out`.
3. Оставить готовый пример или вставить:

```json
{
  "title": "Guitar combo amp",
  "category": "amplification"
}
```

4. Нажать `Execute`.

Что говорить:

> Это пример CRUD-операции create. Аналогично реализованы чтение, обновление и удаление для основных сущностей.

### 2:35-4:00 - Бронирование и проверка конфликта

Что нажимать:

1. Открыть `POST /api/v1/bookings`.
2. Нажать `Try it out`.
3. Использовать готовый пример:

```json
{
  "room_id": 1,
  "starts_at": "2026-05-01T10:00:00",
  "ends_at": "2026-05-01T12:00:00",
  "participant_count": 4,
  "note": "Rehearsal before concert"
}
```

4. Нажать `Execute`.
5. Показать статус `201` и поле `total_price`.

Что говорить:

> При создании бронирования сервис проверяет, что комната существует, активна, подходит по вместимости, а время не пересекается с уже созданными бронированиями. Стоимость считается автоматически по почасовой ставке комнаты.

Показать ошибку:

1. Не меняя JSON, нажать `Execute` еще раз.
2. Показать ошибку `409`.

Что говорить:

> При повторном запросе на то же время сервис возвращает конфликт. Это значит, что бизнес-правило по пересечению бронирований работает.

### 4:00-5:20 - Бизнес-алгоритм подбора комнаты

Что нажимать:

1. Открыть `POST /api/v1/recommendations/rooms`.
2. Нажать `Try it out`.
3. Использовать пример:

```json
{
  "participants": 4,
  "equipment_ids": [1, 2],
  "starts_after": "2026-05-01T10:00:00",
  "ends_before": "2026-05-01T16:00:00",
  "duration_minutes": 120,
  "max_price": 4000,
  "limit": 3
}
```

4. Нажать `Execute`.
5. Показать массив `suggestions`.

Что говорить:

> Это не обычный CRUD-запрос, а алгоритмическая часть проекта. Сервис перебирает свободные 30-минутные слоты, исключает занятое время, проверяет вместимость, бюджет и нужное оборудование. После этого варианты сортируются по рейтингу `score`, где учитывается цена, запас вместимости и близость времени.

Если до этого уже было создано бронирование на 10:00-12:00, объяснить:

> Так как слот 10:00-12:00 уже занят, алгоритм должен предлагать другие подходящие варианты.

### 5:20-6:10 - Техническое решение в коде

Что показать:

1. Перейти в редактор кода.
2. Открыть файл `app/services/recommendations.py`.
3. Показать функции:
   - `load_candidate_rooms`
   - `load_bookings_by_room`
   - `score_slot`
   - `build_room_suggestions`

Что говорить:

> Я вынес бизнес-логику в сервисный слой, чтобы роутер FastAPI оставался коротким. Здесь отдельно загружаются подходящие комнаты, отдельно бронирования по ним, затем строятся варианты слотов и считается рейтинг. Такой подход проще тестировать и поддерживать.

Дополнительно можно показать:

1. `app/models.py` - таблицы и связи.
2. `app/schemas.py` - Pydantic-схемы и валидация.
3. `app/dependencies.py` - проверка JWT и роли администратора.

### 6:10-6:50 - Тесты и pylint

Что нажимать в терминале:

```bash

```

Что сказать:

Потом выполнить:

```bash

```

Что сказать:

## Частые проблемы во время записи

### Swagger показывает `Failed to fetch`

Причина: локальный сервер не запущен.

Решение:

```bash
source .venv/bin/activate
export ROOMFLOW_SECRET_KEY="local-dev-secret-change-me"
export ROOMFLOW_SEED_DEMO=1
export ROOMFLOW_ADMIN_EMAIL="admin@example.com"
export ROOMFLOW_ADMIN_PASSWORD="StrongPass123"
uvicorn app.main:app --reload
```

### После Authorize запросы все равно возвращают 401

1. Нажать `Authorize`.
2. Проверить username и password.
3. Поля `client_id` и `client_secret` оставить пустыми.
4. Нажать `Authorize`, затем `Close`.

### Повторное бронирование возвращает 409

Это нормально. Так демонстрируется защита от пересечения бронирований.

### Комнаты в облаке пустые

Обновить страницу и выполнить `GET /api/v1/rooms` еще раз. Облачная версия использует временную SQLite-БД и автоматически добавляет демо-данные при старте функции.
